## critical

- `app/wave_orchestrator.go:64-66` — `UpdatePlan` only replaces the plan but does NOT transition state from `WaveStateElaborating` to `WaveStateIdle`, does NOT reset `currentWave`, and does NOT clear `taskStates`. The `orchestration/engine.go:97-102` version correctly does all three. The app version is the one actually used at runtime. Currently works by accident because `StartNextWave` lacks the elaborating guard (see below), so it proceeds from `WaveStateElaborating` → `WaveStateRunning` without error. But any future code checking `orch.State()` after `UpdatePlan` would see stale `WaveStateElaborating`, and adding the missing guard to `StartNextWave` would break the signal handling path. Fix: sync the app version with the orchestration version — add `o.state = WaveStateIdle`, `o.currentWave = 0`, `o.taskStates = make(map[int]taskStatus)`.

- `app/wave_orchestrator.go:103-108` — `StartNextWave` is missing the `WaveStateElaborating` guard present in `orchestration/engine.go:107-109`. Without this guard, `StartNextWave` can proceed while the elaborator is still running if anything calls it before the signal handler fires. After fixing `UpdatePlan` above, this guard becomes essential — without it, `startNextWave` called after `UpdatePlan` (which now resets to Idle) would proceed from `WaveStateIdle` which is correct, but the guard is still needed to prevent accidental wave starts during active elaboration. Fix: add `if o.state == WaveStateElaborating { return nil }` at the top of `StartNextWave`, matching the orchestration package.
